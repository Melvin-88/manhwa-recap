# Skill: Prompt Compiler

## Purpose
Перетворює Visual Shot Package + дані реєстру в конкретний промпт для обраного генератора.

## Input
- `04-visual-shot-package.json`
- `registries/style-registry.json` для `generator_config` (поле `generation_backend`)

## Output
`05-prompt-package.json`:
```json
{
  "episode_id": "ep01",
  "prompt_packages": [
    {
      "shot_id": "SHOT_001",
      "prompt_text": "",
      "reference_images": [],
      "generator_globals": {},
      "compiler_flags": {}
    }
  ]
}
```

## Rules
- Вирішує конфлікти між registry-даними (напр. якщо освітлення сцени суперечить дефолтному освітленню локації — сцена має пріоритет).
- Будує промпт під `generation_backend` зі `style-registry.json` (`claude-mcp-media` за замовчуванням для пілоту; `fallback_model` — задокументована, неактивна альтернатива, перехід на яку не вимагає змін цього skill, лише перемикання `generation_backend`).

## Формат зображення: розкадровка за принципами манги/манхви (обов'язкові критерії)

Джерело принципів: усталені практики коміксо-манга розкадровки (Scott McCloud, "Understanding Comics" — типи переходів між панелями, компресія/декомпресія часу; практики манхви/вебтуну щодо варіативності планів і композиції кадру).

Орієнтир — формат популярних відео-переказів манхви: одне зображення на шот (1 шот = 1 файл `SHOT_NNN.png`, кількість файлів не змінюється), але **розкладка всередині нього — не довільний вибір художника, а результат обов'язкової перевірки за критеріями нижче**, яку Prompt Compiler проходить для КОЖНОГО шоту без винятку.

### Крок 1: скільки панелей — обов'язкове рішення
Шот МУСИТЬ бути розбитий на **2-3 панелі**, якщо виконується хоча б одна умова:
- `emotional_beat`/`summary` описує зміну стану чи реакцію (до → після, дія → наслідок) — а не єдиний статичний момент.
- Дія має початок-середину-кінець (напр. кидок → удар → результат) — послідовні панелі за принципом "action-to-action"/"moment-to-moment" (McCloud).
- У кадрі ≥2 персонажі, що взаємодіють (розмова, конфлікт, наказ) — панель встановлення сцени + панель реакції/крупного плану.
- Є репліка чи озвучений момент, що потребує показати і мовця, і слухача.

Одна панель на весь кадр (без розбивки) допустима ЛИШЕ якщо:
- Це чисто встановлюючий (establishing) широкий кадр локації/масштабу без дії й діалогу (напр. панорама шахти) — навмисний "decompressed" момент, splash-кадр; використовувати не частіше ~1 разу на кожні 15-20 шотів, інакше епізод втрачає ритм.
- Це один статичний емоційний удар без вторинної інформації (погляд, поза, тиша) — навмисна пауза для контрасту з динамічнішими сусідніми кадрами.

Це рішення фіксується в `compiler_flags.panel_layout` кожного шоту: `"single"` | `"2-panel"` | `"3-panel"`. Жоден шот не залишається без явного значення цього поля.

### Крок 2: технічна вимога, щоб панелі дійсно рендерились як панелі
Для будь-якого шоту з `panel_layout` ≠ `"single"` промпт ОБОВ'ЯЗКОВО містить фіксовану фразу **"{N}-panel manga page layout with clearly visible black panel borders and white gutters separating each panel (not a single continuous illustration)"** — саме цим текстом, а не лише нумерацією "Panel 1: ... Panel 2: ...". Спостережений дефект: без цієї явної вказівки на межі/ринви генератор регулярно ігнорує номери панелей і малює суцільну повнокадрову ілюстрацію замість розкадровки, навіть коли в тексті промпту є "Panel 1/2/3".

### Крок 3: варіативність "камери" між панелями одного шоту
Кожна панель у межах шоту явно вказує СВІЙ тип кадру (ECU/CU/MS/Full/Wide/Aerial) та за потреби кут (low/high/bird's-eye/worm's-eye/OTS/dutch) з `camera-registry.json`, і жодні дві сусідні панелі одного шоту не повторюють ідентичний тип — контраст дистанцій, властивий професійній розкадровці, а не монотонна послідовність однакових планів.

### Крок 4: композиція кожної окремої панелі
- Правило третин: очі/обличчя персонажа — на перетині ліній сітки 3×3, не в геометричному центрі кадру.
- Headroom (простір над головою) відповідає настрою сцени: багато простору — вразливість/самотність/масштаб; мало — напруга/тиснення.
- "Looking room": якщо персонаж дивиться чи рухається в певному напрямку — лишити негативний простір саме в цьому напрямку кадру.
- Форма/розмір панелі відображає темп: домінуюча велика панель = уповільнення, драматична пауза; кілька дрібних панелей поспіль = прискорення, напруга бою.

### Спіч-бабли й текст
- **Спіч-бабли — лише форма, без тексту всередині.** Якщо в шоті є репліка чи озвучений момент — промпт додає порожній (empty, blank, no text inside) спіч-бабл біля відповідного персонажа, щоб було зрозуміло хто говорить, але не вписує сам текст репліки текстом у зображення (AI-генератори ненадійно рендерять вбудований текст — вихід зазвичай нечитабельна абракадабра).
- **Звукові ефекти (onomatopoeia) — за замовчуванням не використовувати.** Раніше спроби вписати короткі вигуки (WHOOSH, CLANK) регулярно призводили або до незапитаного додаткового тексту в кадрі, або до спотвореного/нечитабельного напису — обидва є QA-дефектами. Використовувати онопатопею лише як свідомий виняток за окремим рішенням, не за замовчуванням.
- Джерело деталей для панелей — те саме, що і для решти пайплайну: `summary`/`emotional_beat` з `01-master-narrative.json`, ніяких нових вигаданих деталей дії.
