# Skill: Prompt Compiler

## Purpose
Turns a Visual Shot Package plus registry data into a concrete prompt for the configured generator.

## Input
- `04-visual-shot-package.json`
- `registries/style-registry.json` for `generator_config` (the `generation_backend` field)

## Output
`05-prompt-package.json`:
```json
{
  "episode_id": "ep01",
  "prompt_packages": [
    {
      "shot_id": "SHOT_001",
      "prompt_text": "",
      "reference_images": [],
      "generator_globals": {},
      "compiler_flags": {}
    }
  ]
}
```

## Rules
- Resolves conflicts between registry data (e.g. if a scene's lighting contradicts the location's default lighting, the scene wins).
- Builds the prompt for the `generation_backend` in `style-registry.json` (`claude-mcp-media` by default for the pilot; `fallback_model` is a documented, inactive alternative — switching to it requires no changes to this skill, only flipping `generation_backend`).
- **Shot-to-shot continuity — read the previous shot's ending state before compiling this one.** When two consecutive shots depict one continuous physical action split across shot boundaries (a fall, a chase, an escape, anything with real-world momentum), check what physical state the previous shot's prompt ended on, and make sure this shot's opening beat picks up from exactly that state — don't skip an implied physical step that a viewer would expect to see. Observed failure mode: one shot ended with the protagonist mid-water, still swimming/surfacing after a fall into a lake; the very next shot opened with him already standing calmly and dry on the shore, with no panel showing him climbing out — the implied "getting out of the water" beat was skipped entirely, so the sequence read as a disconnected jump cut rather than one continuous escape. Fix: when the gap between two shots' physical states would require the viewer to mentally fill in a nontrivial action, add that action as an explicit panel/beat in the compiled prompt (here: a panel of him hauling himself out of the water, dripping, before the calmer standing beat) rather than silently skipping to the resolved state.

## Image Format: Manga/Manhwa-Style Paneling (Mandatory Criteria)

Source of principles: established comic/manga paneling practice (Scott McCloud, "Understanding Comics" — panel-transition types, compression/decompression of time; manhwa/webtoon practice around shot-distance variety and panel composition).

Reference point: the format used by popular manhwa-recap videos — one image file per shot (1 shot = 1 `SHOT_NNN.png` file, the file count never changes), but **the layout inside that file is never an arbitrary artistic choice — it is the result of the mandatory decision procedure below**, which Prompt Compiler runs for EVERY shot without exception.

### Step 1: How Many Panels — a Mandatory Decision
A shot MUST be split into **2-3 panels** if at least one of these holds:
- The `emotional_beat`/`summary` describes a state change or reaction (before → after, action → consequence) rather than a single static moment.
- The action has a beginning-middle-end (e.g. wind-up → impact → result) — sequential panels following the "action-to-action"/"moment-to-moment" principle (McCloud).
- The shot has ≥2 characters interacting (conversation, conflict, a command) — an establishing panel plus a reaction/close-up panel.
- There is a line of dialogue or narrated moment that needs to show both the speaker and the listener.

A single panel for the whole shot (no split) is allowed ONLY if:
- It is a purely establishing wide shot of a location/scale with no action or dialogue (e.g. a panoramic view) — a deliberate "decompressed" beat, a splash panel.
- It is a single static emotional beat with no secondary information (a look, a pose, silence) — a deliberate pause for contrast against more dynamic neighboring shots.

**"2-3 panels" is a range, not a synonym for "2":** re-run the full Step 1 test for every shot independently — do not carry over the previous shot's panel count as a default. If a shot's action has three or more distinct sequential beats (e.g. wind-up → impact → reaction, or three separate characters/techniques that each need their own panel), it MUST get `"3-panel"`, not `"2-panel"` compressed to fit. Observed failure mode: across a long run of consecutive shots, every single one was marked `"2-panel"` regardless of content — including shots with a clear three-beat action (a strike, its impact, and a reaction) — because "2-panel" was being treated as a safe default that technically satisfies "must be split into 2-3 panels" rather than re-deriving the actual count from the shot's content each time. Track the panel-count distribution across the episode the same way the frequency cap tracks `"single"`: a healthy episode has genuine variety (some `"3-panel"` shots, not zero), and a stretch of 20+ shots with no `"3-panel"` at all is itself a signal to re-check Step 1 rather than continue the pattern.

**Hard frequency cap (a mandatory check, not a guideline):** before marking a shot `"single"`, count how many shots ago `panel_layout: "single"` last appeared in this episode's already-compiled `05-prompt-package.json` (including the current partial batch). If it was fewer than **15 shots ago** — even if the shot formally meets the criteria above — it MUST get `"2-panel"` instead (split the same visual moment into an establishing panel plus a detail/reaction panel), not `"single"`. This rule prevents single-panel shots from clustering together or every-other-shot; without it, an aggregate audit found nearly a third of an early shot run marked `"single"` against a ~5-7% target, because per-shot QA only checks whether the rendered image matches its own declared layout — it never checks the aggregate frequency across the episode.

This decision is recorded in each shot's `compiler_flags.panel_layout`: `"single"` | `"2-panel"` | `"3-panel"`. No shot is left without an explicit value for this field.

### Step 2: The Technical Requirement for Panels to Actually Render as Panels
For any shot with `panel_layout` ≠ `"single"`, the prompt MUST include the fixed phrase **"{N}-panel manga page layout with clearly visible black panel borders and white gutters separating each panel (not a single continuous illustration)"** — using exactly this wording, not just numbering like "Panel 1: ... Panel 2: ...". Observed failure mode: without this explicit border/gutter instruction, the generator regularly ignores panel numbering and renders one continuous full-frame illustration instead of a paneled layout, even when the prompt text contains "Panel 1/2/3".

### Step 3: Camera Variety Across Panels Within One Shot
Each panel within a shot explicitly states its own shot distance (ECU/CU/MS/Full/Wide/Aerial) and, where relevant, angle (low/high/bird's-eye/worm's-eye/OTS/dutch) from `camera-registry.json`, and no two adjacent panels within the same shot repeat an identical shot type — the distance contrast that characterizes professional paneling, not a monotone sequence of identical framings.

### Step 4: Composition of Each Individual Panel
- Rule of thirds: a character's eyes/face sit on a 3×3 grid intersection, not the geometric center of the frame.
- Headroom matches the scene's mood: generous headroom = vulnerability/isolation/scale; tight headroom = tension/pressure.
- "Looking room": if a character is looking or moving in a given direction, leave negative space in that direction.
- Panel shape/size reflects pacing: one dominant large panel = a slow-down, a dramatic pause; several small panels in a row = acceleration, action-sequence tension.

### Character Disambiguation: Reference Images Cannot Be Left to the Generator's Discretion
When a shot contains both a registered character (whose `reference_images` are passed in) and **a different, unnamed/background** character (e.g. "some other prisoner," "a guard," "a passerby"), the generator regularly applies the reference image's face to EVERY human figure in the frame unless the prompt text explicitly describes the second character as visibly different. Observed failure mode: a shot depicting a registered character witnessing something happen to an unrelated background character rendered both with the same face, because nothing in the prompt distinguished them — even though the narrative made clear they were two different people.

Mandatory rule: if a shot contains a registered character TOGETHER WITH an unnamed/background character of a different identity, the prompt MUST explicitly describe that second character's appearance (age, build, hair color/style, distinguishing facial features) and add a phrase along the lines of **"an unnamed [role] — [specific distinguishing features] — entirely different in appearance from [registered character's name], not the same person"**. Never rely on an absent description meaning "any person" — the generator treats a gap as "reuse the same reference."

### Combat/Conflict Shots: Always State Who the Opponent Is
When a shot depicts a registered character fighting, threatening, or opposing someone, the prompt must make the opponent's identity and relationship explicit — this is a stricter case of the disambiguation rule above, because getting it wrong here doesn't just blur two background faces, it produces a scene that is narratively impossible (an ally attacking an ally) and directly contradicts the story context. Observed failure mode: a shot's prompt described the protagonist swinging a weapon "against a guard" with no further detail, while the shot's reference images included both the protagonist and a registered ally (needed for that same shot's later panel, showing the ally elsewhere in the group) — the generator applied the ally's face and hairstyle to the undescribed "guard," rendering what reads as the protagonist attacking his own ally, even though the narrative has them fleeing side by side and the only plausible opponent is an enemy guard.

Mandatory rule: for any panel showing conflict or combat, either (a) name the opponent's registered character_id and pass their reference image if they are genuinely a known character, or (b) explicitly describe the opponent's appearance as visibly distinct from every registered character whose reference image is included anywhere in that shot's generation call, and add an explicit relationship-clarifying phrase, e.g. "an enemy guard — bald, clean-shaven, heavier build — entirely different in appearance from [ally's name], an opponent, not an ally." When one shot's panels mix combat (protagonist vs. unnamed enemy) with a separate panel featuring a registered ally elsewhere in the scene, describe the antagonist thoroughly enough in the combat panel's own text that no registered reference bleeds into it — do not assume the panel split alone keeps them apart.

### Consistency of Unnamed Recurring Background Archetypes
If a shot contains a recurring unnamed background archetype recorded in `style-registry.json` → `recurring_background_archetypes` (e.g. a recurring guard/soldier/crowd type), the prompt MUST insert that entry's `visual_description` verbatim, rather than re-describing the archetype from scratch each time. Observed failure mode: without a fixed description, a recurring background archetype was rendered with wildly different styling from shot to shot (e.g. drifting between two unrelated visual eras or aesthetics) depending on how a given prompt call happened to phrase it. If a shot needs a new recurring background archetype that isn't in the registry yet, add an entry to `recurring_background_archetypes` first (analogous to Registry Builder for CHAR/CRE), then use it in the prompt.

### Environment Completeness
Every shot's prompt must name the specific location it is set in — pulled from the shot's established scene/location context — regardless of whether that location is an enclosed interior or an outdoor setting. If a shot takes place inside a vehicle, room, or other enclosed environment, the prompt explicitly names the visible elements of that environment (for a vehicle: dashboard, doors, windows, seats — not just a control the character is holding against an empty background). Observed failure mode #1: a shot set inside a vehicle rendered the character holding a steering wheel against an empty background with no visible car body — reading as if the character were sitting in open air rather than inside a car. Observed failure mode #2: a shot's prompt named characters and action but never named the environment at all; the generator filled in a flat, featureless background with no location cues, breaking continuity with the clearly-established setting of the surrounding shots. Observed failure mode #3: a shot in a sequence of consecutive forest shots omitted the environment phrase for just that one shot, and the generator defaulted to a generic indoor stone-masonry interior instead — directly contradicting both the story's established setting and every neighboring shot in the same batch. When a shot is part of a run of shots sharing one location, repeat that location's key visual descriptors in every shot's prompt — never assume continuity carries over from a previous shot's prompt to this one.

### Recurring Creature/Character Reference Sheets Establish Design, Not Pose
A reference sheet (turnaround/model sheet) exists to lock a character's or creature's face, proportions, and color palette — it is not a pose to copy into an action shot. Observed failure mode: a recurring creature's reference sheet showed it in a neutral, symmetrical standing pose (arms at sides, feet planted evenly, front-facing) for the purpose of establishing its design; multiple later shots depicting that creature mid-chase or mid-hunt rendered it in that same static, symmetrical standing pose with no motion at all, making an active pursuit scene look like the creature was simply parked in the background. Mandatory rule: any shot showing a recurring character/creature in motion or action MUST explicitly describe a dynamic pose appropriate to that action (mid-stride, weight shifted onto one leg, torso leaning in the direction of movement, limbs bent rather than symmetrically resting) and should include an explicit instruction such as "a dynamic action pose, not the neutral reference-sheet stance." Never assume passing the reference image alone is enough to convey motion — describe the motion in words every time.

### Visual Family Cohesion Across Different Forms of the Same Species/Faction
When a story establishes that a species, faction, or enemy type has multiple visually distinct forms (e.g. different body plans for different power tiers of the same monster species), each form's prompt must still carry a shared, explicit visual motif tying it to the others — even when the individual body plans differ completely — so a viewer with no narrative context can still recognize them as the same category of threat. Observed failure mode: one member of an enemy species was a small quadruped beast (fangs, single eye) and another was a towering armored humanoid; both were lore-accurate distinct forms of the same species, but with no shared visual language between their prompts, they rendered as two completely unrelated monster designs, undermining the sense that they belonged to a single recognizable enemy type. Fix: identify the species' unifying trait (a color palette, a material, an aura/particle effect, a marking) and state it explicitly in every form's prompt, regardless of how different the underlying body plan is. Register each distinct form in `creature-registry.json` with its own entry (see Registry Builder) and copy that entry's `visual_description` — including the shared motif — into the prompt verbatim, the same way `recurring_background_archetypes` entries are reused.

### Both Hands Must Be Explicitly Accounted For — Not Just the Active One
Any shot description that specifies what ONE of a character's hands is doing (holding, punching, gripping, cradling an object) while leaving the OTHER hand/arm's pose unstated is an invitation for the generator to invent an extra hand rather than simply leaving the other arm idle. This isn't limited to action/impact shots — it recurred in a calm, static single-panel pose just as readily as in a punching close-up. Observed failure mode #1: a close-up punching panel rendered the correct fist making contact plus a second, smaller disconnected fist lower in the frame with no continuous arm connecting it to the character's body — read as shadow/texture near the point of impact rather than an anatomy error on a holistic look (see Image QA's hand/limb check protocol). Observed failure mode #2: a quiet single-panel shot's prompt described one hand cradling a glowing object and "a tendril flowing into his hand," but never stated what the character's other arm was doing — the generator invented a third hand touching the character's chest in addition to a separate hand resting near his knee, effectively giving the same arm two endpoints.

Mandatory rule: whenever a shot's prompt specifies an action or pose for one of a character's hands, it must also explicitly state what the other hand/arm is doing (resting on a knee, at his side, out of frame, etc.) and add "exactly two hands/arms total, no extra, duplicated, or floating third hand." Never leave a second hand's pose to be inferred — an unstated hand is exactly what the generator fills in with an invented extra limb.

### Realism and Dynamic Motion in Action Beats
Whenever a shot depicts the moment during or immediately after a physical action (a throw, impact, landing, block, running stride), the prompt must describe physics-consistent dynamic follow-through — weight shift onto the driving leg/arm, muscle tension, an off-balance or still-recoiling posture, trailing motion lines or settling debris — rather than a neutral, flat, relaxed standing pose. A default "calm standing" description reads as static and lifeless immediately after a high-intensity action, breaking the sense of physical continuity the moment implies. Observed failure mode: a panel depicting the immediate aftermath of an extremely forceful spear throw (the spear still embedded in freshly cracked rock, dust falling) rendered the protagonist in a calm, upright, neutral stance with no visible physical reaction to his own effort — no weight shift, no lingering tension, no follow-through in the throwing arm — reading as flat and posed rather than as the dynamic, still-settling conclusion of a powerful action. Fix: explicitly describe the character's body still reacting to the just-completed action (weight still loaded on the throwing-side leg, the throwing arm still extended rather than relaxed at his side, visible tension or heavy breathing) instead of defaulting to a settled, neutral pose. This applies to every future action-adjacent shot, not just impact panels — err toward describing the specific physical state a real body would be in one second after the depicted action, never a generic idle stance.

### One Panel = One Moment in Time Per Character
A single, undivided panel must show each character in exactly one pose from one instant — never two separate temporal states of the same character (e.g., a mid-fall pose and an already-landed/recovered pose) rendered together within the same continuous panel, since this reads as two different people rather than one character's motion. Observed failure mode: a panel meant to depict a character flying through the air mid-impact instead rendered that same character twice within one undivided panel — once diving into water, and again already sitting recovered against a tree elsewhere in the same frame — creating the visual impression of two separate characters rather than a single mid-flight moment. Fix: state explicitly, e.g. "exactly one instance of the protagonist in this panel, showing only [the single described instant] — do not depict him a second time already in a later/different state anywhere else in this same panel." If a shot genuinely needs to show a temporal before/after within one beat, split it into a proper multi-panel layout (see Step 1) rather than relying on a single panel to imply two states.

### Dead/Inert Remains Must Not Glow or Show Signs of Life
When a shot's environment includes the corpse, skeleton, or other remains of a creature explicitly established as dead/inert (as opposed to the same species' living members elsewhere in the story), the prompt must explicitly state that any features which glow or emit light on the living version — eyes, markings, cores — are dark, empty, and lightless on the remains. Never assume "skeleton"/"corpse"/"remains" alone is enough to convey deadness; left unspecified, the generator will often default to rendering glowing eye sockets exactly as if the creature were alive and watching, which both breaks the "long dead" narrative fact and (if the living version has a specific registered eye count/color) mismatches that registered design too. Observed failure mode: a shot describing "the same cavern lair with the abyssal skeleton" (no further qualifier) rendered the skull with two glowing red eye sockets, even though the creature was established as dead and its living registry entry specifies a single glowing orange eye, not two red ones. Fix: state explicitly, e.g. "the skeleton's eye sockets are completely empty, dark, and lightless — long-dead inert bone with no glow of any kind."

### Panels: Borders and Count — No Exceptions
Every panel without exception (including the last/edge one) MUST have a visible border on **all four sides** — top, bottom, left, AND right — even the side(s) that touch the outer boundary of the image itself. Observed failure mode #1: a multi-panel shot rendered its final panel with no border on the bottom, bleeding into the image edge, while the earlier panels were clearly framed. Observed failure mode #2: in a 2-panel side-by-side layout, the rightmost panel had a border on its top, left, and bottom, but the right edge — which coincides with the outer edge of the canvas — had no border line at all, bleeding straight into the page margin; this is easy to miss because the other three sides look correct at a glance. The prompt's border instruction must say explicitly that the panel touching the canvas edge still needs a border drawn on that edge, not just the internal gutter sides. Likewise, the gutter (white space) between panels must be a thin, ordinary-width gap — not a wide white band that itself reads as an empty panel (observed failure mode: a wide gap between two panels was mistaken for a third, blank panel). **The reverse failure is equally a defect and just as easy to miss: the gutter itself containing actual image content.** Observed failure mode #3: a declared 2-panel shot rendered a third, narrow bordered sliver wedged between the two real panels, filled with a duplicated fragment of panel 1's background — effectively three vertical strips instead of two clean panels, even though each individual panel's own four-sided border looked fine in isolation. Observed failure mode #4: a held object (a thrown spear) drawn in panel 1 physically crossed that panel's own border line and extended into the gutter itself, and the gutter was filled with a continuation of the panel's background color rather than being blank — this was initially and wrongly waved through as "a minor stylistic variance" because the border line was technically still drawn and visible, but a border that content visibly crosses is not functioning as a border. The prompt must state explicitly that the gutter is a "single plain white gap with absolutely no image content, no duplicated background, no leftover artwork inside it" AND that "no object or body part from either panel may cross or extend past its own panel's border into the gutter or into the neighboring panel" — a border is only correct if every element of the illustration stops exactly at it, not merely if a line is drawn there. Never assume "gutter" alone rules out the generator filling that space with a stray extra strip of art or letting content bleed across the line.

### Every Panel Must Carry Its Own Distinct Information
No panel may substantively repeat what another panel in the same shot already shows without adding a new detail or angle that changes the meaning. Mirror/abstract compositional devices (e.g. a reflection in water) are acceptable only when they clearly and unambiguously convey a specific described detail (a face, an expression) — not as a purely decorative repeat of the same pose or object. Observed failure mode: a panel intended as a close-up on a specific body part instead rendered a symmetric full-body mirror reflection that didn't read as a meaningful detail.

### Size of Plot-Critical Objects — Always Anchor to Body/Hand Scale
When a shot describes an object whose size matters for the plot (something swallowed or pinched between fingers, a small artifact, etc.), the prompt MUST specify a concrete size relative to the character's body/hand (e.g. "no bigger than a thumb tip," "about the size of a grape or a small marble") rather than an abstract descriptor like "a small X" or "a good-sized X". Observed failure mode: two separate shots both describing a "small" or "modestly sized" object instead rendered it as roughly half the size of the character's palm — inconsistent with both the source description and the physical plausibility of the described action (pinching between two fingers, swallowing in one motion).

### A Character's Actions Must Not Contradict Their Established Strategy/Motivation
If the master narrative or a detailed beat explicitly establishes a character's motivation (e.g. "avoid drawing attention," "pretend to be weak"), no shot may depict a public, conspicuous display of the opposite — even if the source material mentions the character's growing abilities in the abstract. Render growing strength/ability through hidden, non-public detail (daily hard labor visibly costs less effort, less strain on the face, a furtive glance to check no one noticed) rather than through a showy, openly demonstrated feat in front of others. Observed failure mode: a shot meant to convey a character's quietly growing strength instead depicted an overt public feat that was never described in the source material and directly contradicted that character's established "stay unnoticed" strategy.

### Speech Bubbles and Text
- **Speech bubbles are shape only, never text inside.** If a shot includes a line of dialogue or a narrated moment, the prompt adds an empty (blank, no text inside) speech bubble near the relevant character so it's clear who is speaking, but never writes the actual line as text in the image (AI generators render embedded text unreliably — the output is usually illegible gibberish).
- **Sound effects (onomatopoeia) are not used by default.** Earlier attempts to write short exclamations (e.g. impact/whoosh sounds) into prompts regularly produced either unrequested extra text in the frame or garbled/illegible lettering — both are QA defects. Use onomatopoeia only as a deliberate, case-by-case exception, never as a default.
- The source of detail for panels is the same as for the rest of the pipeline: `summary`/`emotional_beat` from `01-master-narrative.json` — never new invented action detail.
