# Skill: Source Ingestion

## Purpose
Перший етап пайплайну. Перетворює джерело (веб-новела/манхва) у структуровані факти — персонажів, світ, сюжетні події — без збереження оригінального тексту дослівно. Працює на рівні всього твору: факти накопичуються в єдиному master-файлі поступово, батчами розділів, а не наново для кожного епізоду. Це і є момент, де починається трансформація/стиснення історії.

## Input
- `story_slug` — визначає фолдер `stories/<story_slug>/`, у якому живуть усі матеріали цього твору (реєстри, епізоди, референс-зображення персонажів); `work_slug` нижче — те саме значення
- Кореневий URL твору + перелік уже оброблених розділів (щоб знати, з якого розділу продовжити)
- Розмір батчу на цей прохід (наприклад "розділи 1-15") — узгоджується з людиною перед запуском, не масовий скрейпінг усього твору одним викликом
- Наявний `stories/<story_slug>/source-material/master-extract.json`, якщо це не перший прохід (нові факти додаються до існуючих, не перезаписують їх)

## Output
`stories/<story_slug>/source-material/master-extract.json` — накопичувальний факт-стор усього твору:
```json
{
  "work_slug": "the-greatest-heretic",
  "source_root_url": "https://example.com/work",
  "chapters_ingested": [
    {"chapter": "v1c1", "url": "https://example.com/chapter-1"}
  ],
  "characters_extracted": [
    {
      "source_name": "",
      "adapted_name": "",
      "role": "protagonist | antagonist | supporting",
      "key_traits": [],
      "first_appeared_chapter": "v1c1",
      "auto_generated_fields": []
    }
  ],
  "world_facts": [
    {"fact": "", "first_appeared_chapter": "v1c1"}
  ],
  "plot_beats": [
    {"beat_id": "BEAT_001", "chapter": "v1c1", "summary": "", "characters_involved": []}
  ],
  "adaptation_notes": {
    "renamed": [],
    "compressed": [],
    "changed": []
  }
}
```

Episode-level `00-source-extract.json` більше не є окремим прогоном ingestion — це **зріз** з `master-extract.json` за діапазоном `plot_beats`/розділів, які припадають на конкретний епізод (шлях відносний до кореня `stories/<story_slug>/`):
```json
{
  "episode_id": "ep01",
  "master_extract_ref": "source-material/master-extract.json",
  "beat_range": ["BEAT_001", "BEAT_009"],
  "chapters_covered": ["v1c1", "v1c2"]
}
```

## Rules
- **Оригінальний текст джерела ніколи не зберігається дослівно в жодному файлі репозиторію.** Вихід — факти й короткі власні формулювання, не перефразовані речення оригіналу.
- Вихід має бути суттєво коротшим за джерело.
- Кожен прохід обробляє погоджений з людиною батч розділів (наприклад 10-15), не весь твір одним викликом — практичне обмеження через навантаження на джерело, не тільки юридичне.
- Потребує можливості читання веб-сторінки (web fetch / browser tool залежно від агента, що виконує цей skill).
- **Прогалини заповнюються автоматично.** Якщо факт (наприклад зовнішність персонажа) ще не описаний у прочитаних розділах — Source Ingestion сам генерує правдоподібну деталь і позначає її в `auto_generated_fields`, а не зупиняється з питанням. Причина: рано в творі невідомо, де саме проявиться та чи інша деталь, тож блокувати процес на кожному нез'ясованому полі недоцільно.
- Людина отримує підсумок для підтвердження ПІСЛЯ завершення батчу (загальна картина персонажів/світу/сюжету), а не по одному питанню на розділ.
- `adaptation_notes` обов'язкові й конкретні — основа чекпоінта підтвердження після батчу.
- Персонажі/локації з `characters_extracted` передаються в Registry Builder для створення unlocked-записів із `source_name` та `adapted_name`; поля, позначені в `auto_generated_fields`, можна перезаписати пізнішими розділами, якщо джерело дає точнішу інформацію.
