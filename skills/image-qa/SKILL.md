# Skill: Image QA

## Purpose
Останній автоматичний контроль якості вже згенерованих зображень (кадри епізоду й референс-аркуші персонажів/істот/пропсів) — перш ніж кадр вважається прийнятим у пайплайн. Ловить дефекти, які Image Director і Registry Builder не перевіряють: анатомічні помилки, зламану консистентність персонажа, технічні артефакти, невідповідність сцені. При провалі — сам виправляє неоднозначну частину промпту й перегенеровує, з жорстким лімітом спроб, щоб не спалювати токени/кредити на нескінченні retry.

## Input
- Для кадрів епізоду: `06-generation-log.json` (щойно записані Image Director записи з `asset_path`) + відповідний `05-prompt-package.json` (оригінальний `prompt_text`) + `03-storyboard.json` (очікувані `characters_in_frame`/`scene_id` для перевірки відповідності сцені)
- Для референс-аркушів: щойно згенерований `reference_sheet_path` з `character-registry.json` / `creature-registry.json` / `prop-registry.json`

## Output
Оновлює вхідний лог **in place** (той самий `06-generation-log.json`, або еквівалентний запис для референс-аркуша), додаючи QA-поля до кожного запису:
```json
{
  "shot_id": "SHOT_004",
  "asset_path": "episodes/ep01/generated/SHOT_004.png",
  "generated_at": "...",
  "generator_call": "...",
  "flagged_for_review": false,
  "qa_status": "passed | regenerated | failed_flagged",
  "qa_attempts": 1,
  "qa_notes": "",
  "revised_prompt_used": ""
}
```
`flagged_for_review` — єдине поле, яке читають подальші кроки пайплайну (Assembly, чекпоінти людини); саме Image QA є єдиним джерелом істини для його фінального значення. Image Director і Registry Builder більше не виносять власного вердикту про якість — лише генерують і завантажують.

## Rules

### Перевірка (одним проходом зору по зображенню, 4 категорії)
1. **Анатомічні помилки** — зайві/злиті кінцівки, голови, тулуби, неможливі пози.
2. **Консистентність персонажа** — обличчя/зачіска/одяг у кадрі відповідають актуальному запису в `appearance_states` персонажа (див. правило вибору версії в Image Director) чи референс-аркушу істоти/пропсу.
3. **Технічні артефакти** — текст, watermark, логотипи, що просочилися з референсів; спотворені/розмиті краї, що виглядають як збій генерації (не як навмисний стиль). **Виняток:** запитаний у `prompt_text` короткий звуковий ефект (onomatopoeia — WHOOSH, CLANK тощо) у кадрі з різкою дією — навмисний елемент стилю (див. правило Prompt Compiler), не дефект. Так само порожній (без тексту) спіч-бабл — навмисний елемент, не дефект. Дефектом лишається: будь-який **незапитаний** текст (модель додала SFX/напис, якого не було в промпті), або спіч-бабл, що мав бути порожнім, але модель вписала в нього нечитабельний/спотворений текст.
4. **Відповідність сцені** — персонажі, локація й дія в кадрі відповідають `characters_in_frame`/`composition_goal` зі storyboard (для референс-аркушів — відповідність призначенню: розворот анфас/3-4/профіль + вирази).

Поріг навмисно **нестрогий**: провалює лише явний, недвозначний дефект (як зламана анатомія SHOT_004), а не дрібні стилістичні відхилення чи суб'єктивні художні розбіжності. Мета — не плодити зайві перегенерації.

### Retry (максимум 2 спроби на кадр: 1 оригінал + 1 повторна генерація)
1. Перша невдача → аналізує **причину** дефекту, знаходить конкретну неоднозначну частину `prompt_text`, що ймовірно її спричинила (типовий випадок: опис дії зміни стану — "forced to exchange X for Y", "removes his clothes and puts on" — модель малює обидва стани одночасно замість одного). Переписує **лише цю частину** на однозначний опис одного статичного стану (як правило — кінцевого/актуального стану за сценою), решту тексту промпту й усі `reference_images`/`medias` не чіпає.
2. Викликає генератор повторно з переписаним промптом (той самий модель/референси, що й оригінал).
3. Завантажує новий файл, **перезаписуючи** попередній `asset_path` (немає сенсу тримати відомо-зламаний файл).
4. Проганяє QA повторно на новому зображенні.
5. Якщо друга спроба також провалилась — **зупиняється**, `flagged_for_review: true`, `qa_status: "failed_flagged"`, лишає останній (другий) варіант файлу як best-effort, і пише в `qa_notes` людською мовою який саме дефект лишився — для ручного огляду. Ніколи не робить третю спробу автоматично.
6. Якщо перша спроба одразу пройшла — `qa_status: "passed"`, `qa_attempts: 1`, `revised_prompt_used: ""`.
7. Якщо друга спроба пройшла — `qa_status: "regenerated"`, `qa_attempts: 2`, `revised_prompt_used` містить фактичний текст, яким генерували.

### Межі відповідальності
- Ніколи не змінює сам `05-prompt-package.json` (джерело істини Prompt Compiler) — переписаний промпт існує лише як разовий виклик генератора і записується в `revised_prompt_used` для трасування.
- Ніколи не перепроєктовує персонажа/сцену — лише судить готове зображення і за потреби спрощує формулювання дії в межах уже затвердженого опису кадру.
- Застосовується як до кадрів епізоду, так і до референс-аркушів персонажів/істот/пропсів, які генерує Registry Builder — та сама модель генерації, той самий ризик дефектів.
