# Skill: Image QA

## Purpose
The last automated quality gate on already-generated images (episode frames and character/creature/prop reference sheets) before a frame is accepted into the pipeline. Catches defects that Image Director and Registry Builder don't check: anatomy errors, broken character consistency, technical artifacts, scene mismatches. On failure, fixes the ambiguous part of the prompt itself and regenerates, with a hard retry cap so it never burns tokens/credits on endless retries.

## Input
- For episode frames: `06-generation-log.json` (the Image Director entries just written, with `asset_path`) + the matching `05-prompt-package.json` (the original `prompt_text`) + `03-storyboard.json` (the expected `characters_in_frame`/`scene_id` to check scene match)
- For reference sheets: the just-generated `reference_sheet_path` from `character-registry.json` / `creature-registry.json` / `prop-registry.json`

## Output
Updates the input log **in place** (the same `06-generation-log.json`, or the equivalent entry for a reference sheet), adding QA fields to each entry:
```json
{
  "shot_id": "SHOT_004",
  "asset_path": "episodes/ep01/generated/SHOT_004.png",
  "generated_at": "...",
  "generator_call": "...",
  "flagged_for_review": false,
  "qa_status": "passed | regenerated | failed_flagged",
  "qa_attempts": 1,
  "qa_notes": "",
  "revised_prompt_used": ""
}
```
`flagged_for_review` is the only field later pipeline steps read (Assembly, human checkpoints); Image QA is the single source of truth for its final value. Image Director and Registry Builder no longer render their own quality verdict — they only generate and download.

## Rules

### Checks (one visual pass over the image, 8 categories)
1. **Anatomy errors** — extra/fused limbs, heads, torsos, impossible poses. **Hand/limb check protocol — do this by tracing, not glancing:** for EVERY panel with a visible character, regardless of whether the shot is a dynamic action beat or a calm static pose, trace each visible arm from shoulder to hand and confirm it resolves to exactly the character's actual hand count (normally two) before passing the panel. A holistic glance routinely misses a small extra hand tucked into the composition — it reads as "detail," shadow, or fabric fold at a glance, the same way a missing panel border is easy to miss without checking each side individually (see category 5's border protocol). This check applies just as much to quiet single-panel portrait shots as to punching/gripping close-ups — the failure has occurred in both. Observed failure mode #1: a punching close-up panel showed the correct fist making contact, but a second, smaller disconnected fist/hand appeared lower in the frame with no continuous arm connecting it to the character's body — passed on a first holistic look because attention was on a different check (background/lighting) for that same panel, and the stray hand read as texture/shadow rather than an anatomy error until traced explicitly. Observed failure mode #2: a calm single-panel shot showing a character holding an object in one hand, with the prompt never stating what the other arm was doing, rendered that other arm bifurcating into two separate hands (one at the chest, one resting near a knee) — passed on a first look because the pose read as natural at a glance, and only tracing each arm from shoulder to fingertip individually revealed the second arm had two endpoints. **Also fails a shot whose `prompt_text` describes an action/motion (chasing, fighting, running) but a recurring character/creature is rendered in a static, symmetrical, reference-sheet-like standing pose** — a lifeless "parked" figure in what should be a dynamic scene is as much a defect as broken anatomy, even though no limb is technically malformed (see Prompt Compiler's "Reference Sheets Establish Design, Not Pose" rule).
2. **Character consistency** — the face/hairstyle/outfit in the frame match the character's current `appearance_states` entry (see the version-selection rule in Image Director) or the creature's/prop's reference sheet. **The inverse case is equally a defect:** if the scene, per the storyboard/`prompt_text`, contains a different, unnamed person distinct from the registered character (e.g. "another bystander," "a different member of the crowd"), and the image gives that person the same face/hairstyle as the registered character whose reference was passed in — that's reference bleed (the generator mistakenly applied the reference face to every human figure in the frame); it fails the check the same way broken anatomy would. **Escalate this to a hard fail (not just a note) whenever the bled-in character is cast as an opponent/antagonist against a registered ally** — e.g. a fight scene where the "enemy" visually matches a registered ally's face and hairstyle, producing a scene that reads as the protagonist attacking their own ally. This is a narrative-logic break, not just a visual-consistency slip, and must be caught even if every individual face in isolation looks well-drawn (see Prompt Compiler's "Combat/Conflict Shots" rule).
3. **Technical artifacts** — text, watermarks, logos that leaked in from references; distorted/blurred edges that read as a generation failure (not as a deliberate style choice). **Exception:** a short sound effect (onomatopoeia) requested in `prompt_text` for a sharp-action shot is a deliberate style element (see the Prompt Compiler rule), not a defect. Likewise, an empty (textless) speech bubble is a deliberate element, not a defect. Still a defect: any **unrequested** text (the model added an SFX word or caption that wasn't in the prompt), or a speech bubble that was supposed to stay empty but the model filled with illegible/garbled text.
4. **Scene match** — the characters, location, and action in the frame match the storyboard's `characters_in_frame`/`composition_goal` (for reference sheets: matching the sheet's purpose — front/3-4/profile turnaround + expressions). **A blank/generic/featureless background, or a background that contradicts the shot's established location (e.g. an indoor stone interior rendered for a shot that should be in a forest, matching every neighboring shot), is a scene-match failure** — not a minor stylistic gap. **Also check continuity against the immediately preceding shot** when both depict one continuous physical action (a fall, a chase, an escape): if this shot's opening state silently skips a nontrivial physical step the previous shot's ending implied (e.g. the previous shot ends mid-water and this one opens with the character already dry and standing on shore, with no beat showing him climbing out), that's a continuity failure — flag it the same as a scene-match defect, per Prompt Compiler's "Shot-to-shot continuity" rule.
5. **Panel-layout compliance** — if a shot's `compiler_flags.panel_layout` in `05-prompt-package.json` is `"2-panel"`/`"3-panel"`, the generated image actually shows that many clearly separated panels with visible borders/gutters (not one continuous illustration, regardless of numbered "Panel 1/2/3" text in the prompt). This is a known, recurring generator failure mode, so it's checked as its own category rather than folded into "technical artifacts." **Border check protocol — do this explicitly for every panel, not just a glance:** for each panel, individually confirm all four sides in turn — top, bottom, left, right — including the side(s) that coincide with the outer edge of the canvas. Do not stop checking a panel once three sides look correct; the recurring failure mode is specifically a single missing edge (most often the outer-canvas-facing edge of the last panel in a row), which is easy to miss if the check stops early because "the panel clearly has a border." Also verify the gutter between panels is a thin gap rather than a wide white band that itself reads as a separate empty panel. **Also check the declared panel count against the actual count by zooming into every gutter, not just glancing at the overall layout:** a shot declared as N panels can render as N+1 — a narrow bordered sliver wedged into what should be a blank gutter, filled with a duplicated fragment of the neighboring panel's background. Each individual panel's own borders can look perfectly correct in isolation while this extra sliver still exists between them, so the check must confirm the gutter space itself is blank, not just that each intended panel has four sides. **A border line being present is not sufficient — also confirm nothing crosses it.** Zoom into the gutter and trace it end to end: if any object or body part from a panel's illustration (a held weapon, a limb, hair) visibly extends past that panel's own border line into the gutter, or if the gutter is filled with a continuation of the panel's background color instead of being blank, that is a defect — do not wave it through as "a minor stylistic variance" just because the border line itself is technically drawn. A border only functions as a border if the illustration actually stops at it.
6. **Consistency of unnamed recurring background archetypes** — if the frame contains an archetype recorded in `style-registry.json` → `recurring_background_archetypes` (e.g. a recurring guard/soldier/crowd type), its appearance matches the recorded `visual_description` (the same costume/style in this frame as in every other frame featuring it), rather than a fresh, arbitrary interpretation of the role each time (observed failure mode: a recurring background archetype drifted between two unrelated visual eras/styles across different shots of the same story). **Also applies across different forms of the same registered species/faction in `creature-registry.json`** (e.g. a beast-type and a humanoid-type entry for the same enemy species): check that both carry the shared visual motif recorded in their respective `visual_description` fields, so they read as related even with completely different body plans — two forms of a supposedly single enemy species with no shared visual language between them is a defect, not an acceptable creative variance.
7. **Environment completeness** — if the shot takes place in an enclosed environment (vehicle, room, building interior), the image actually shows that environment's characteristic elements (for a vehicle: the cabin, doors, dashboard — not just a control held against an empty background as if the character were outside the vehicle). **Also check that any remains (corpse/skeleton) explicitly established as dead show no signs of life** — glowing eyes, markings, or light of any kind on a "long-dead" creature's remains is a defect, the same class as a scene contradicting its established location (see Prompt Compiler's "Dead/Inert Remains Must Not Glow" rule); this applies even if the living version of that same creature is registered with its own genuine glowing feature elsewhere.
8. **Each panel carries its own meaning** — no panel is a purely decorative repeat of another panel in the same shot without a new detail. Mirror/abstract compositions (e.g. a reflection in water) are acceptable only when they clearly read as the specific detail described in the prompt (e.g. "a close-up on the face and hands"), not when they turn into a confusing repeat of the same pose.
9. **Realism and dynamic follow-through in action-adjacent poses** — a panel depicting the moment during or immediately after a physical action (throw, impact, landing, block, stride) fails this check if the character is rendered in a calm, neutral, flat standing pose with no physical reaction to the action just performed. Check specifically for: weight shift onto the driving leg/arm, follow-through in the acting limb (not relaxed at the side), visible tension, and any settling debris/motion cues implied by the prompt. Observed failure mode: a panel following an extremely forceful spear throw (embedded in freshly cracked rock) rendered the protagonist standing upright and relaxed, with no trace of the throw's physical toll — reads as static/lifeless rather than as the dynamic aftermath of a powerful action (see Prompt Compiler's "Realism and Dynamic Motion in Action Beats" rule).
10. **One character, one moment, per panel** — a single undivided panel must not render the same character twice in two different temporal states (e.g., mid-fall in one part of the frame and already landed/recovered in another part of the same frame). This reads as two separate characters rather than one character's motion and is a defect distinct from anatomy errors (each individual instance may be anatomically correct in isolation). Observed failure mode: a panel meant to show a character flying through the air mid-impact instead showed that same character twice within the one panel — once diving into water, and again already sitting recovered against a tree elsewhere in the frame (see Prompt Compiler's "One Panel = One Moment in Time Per Character" rule).

The bar is intentionally **lenient**: it fails only a clear, unambiguous defect (broken anatomy, a fully missing panel border on a declared multi-panel shot, a recurring background archetype rendered in an obviously wrong style) — not minor stylistic deviation or subjective artistic disagreement. The goal is not to trigger unnecessary regenerations.

### Batch-Level Check (once, after QA has run on every shot in the batch)
Categories 1-8 above check each shot in isolation — that's not enough to catch a cumulative defect. After finishing QA on every shot in the current batch, compute the share of `compiler_flags.panel_layout: "single"` among all shots in `05-prompt-package.json` that already have an entry in `06-generation-log.json` (the whole episode so far, not just the current batch). If that share exceeds **~1 in 15 shots (~7%)**, that's a violation of the Prompt Compiler frequency cap (see its "Step 1"), not an acceptable QA outcome — report it in the batch summary as a standalone issue (not buried in one shot's `qa_notes`), and don't silently mark it "passed" even if every individual shot is locally correct.

Also check the reverse imbalance: count how many of the last ~20 already-logged shots are `"3-panel"`. Zero `"3-panel"` shots in a 20+ shot stretch is itself a defect signal — it means `"2-panel"` is being used as a default rather than re-derived per shot from its actual content (see Prompt Compiler's "'2-3 panels' is a range" note). Report this the same way: a standalone batch-summary issue, not something to silently pass.

### Retry (max 2 attempts per frame: 1 original + 1 regeneration)
1. On first failure → analyze the **root cause**, find the specific ambiguous part of `prompt_text` that likely caused it (a common case: a description of a state-change action — "forced to exchange X for Y," "removes clothing and puts on Y" — where the model renders both states simultaneously instead of one). Rewrite **only that part** into an unambiguous description of a single static state (typically the final/current state per the scene); leave the rest of the prompt text and all `reference_images`/`medias` untouched.
2. Call the generator again with the rewritten prompt (same model/references as the original).
3. Download the new file, **overwriting** the previous `asset_path` (no value in keeping a known-broken file).
4. Run QA again on the new image.
5. If the second attempt also fails — **stop**, set `flagged_for_review: true`, `qa_status: "failed_flagged"`, keep the last (second) variant as a best-effort file, and write a plain-language description of the remaining defect into `qa_notes` for manual review. Never make a third attempt automatically.
6. If the first attempt passes outright — `qa_status: "passed"`, `qa_attempts: 1`, `revised_prompt_used: ""`.
7. If the second attempt passes — `qa_status: "regenerated"`, `qa_attempts: 2`, `revised_prompt_used` contains the actual text used to generate it.

### Boundaries of Responsibility
- Never modifies `05-prompt-package.json` itself (Prompt Compiler's source of truth) — a rewritten prompt exists only as a one-off generator call and is recorded in `revised_prompt_used` for traceability.
- Never redesigns a character or scene — only judges the finished image and, if needed, simplifies the wording of an action within the already-approved shot description.
- Applies equally to episode frames and to character/creature/prop reference sheets generated by Registry Builder — same generation model, same defect risk.
